# 为什么选择 NgaClient 双层架构（传输层+API 层）

## 背景

NGA 论坛的 HTTP API 存在特殊的编码（GB18030）、多域名故障转移、以及多种请求体格式（JSON、表单、Multipart）的需求。

## 方案对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| 双层架构（NgaClient + NgaApi）✅ | 传输层与业务层解耦、域名故障转移集中一处、请求格式统一处理 | 多一层抽象增加文件数量 |
| 单层（每 API 直接 HTTP 请求） | 代码直观 | 域名重试、编码处理、错误降级在每处都要重复实现 |
| 封装为独立 SDK 模块 | 复用性好 | 构建配置复杂，模块间依赖管理成本高 |

## 决策

`NgaClient` 负责 HTTP 传输、编码解码、域名故障转移；`NgaApi` 封装具体业务接口，调用 `NgaClient` 并利用 taskpool 在后台执行解析。

## 理由

1. 域名故障转移涉及 3+ 个域名的遍历重试，如果分散在各 API 中，维护困难且容易遗漏
2. GB18030 编码解码在传输层统一完成，业务层只处理解析后的对象
3. `NgaApi` 中的 taskpool 并发解析（如 `HtmlParseTask`）可以独立调整不波及传输层

## 代价

- 新增 API 需要在 `NgaApi` 和 `NgaClient` 两个文件中声明接口和实现
- 某些场景下（如 `getHtmlText`）需要绕过 JSON 解析，特例逻辑增加到传输层中
