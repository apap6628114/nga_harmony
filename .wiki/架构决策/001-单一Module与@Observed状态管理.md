# 为什么选择单一 Module + @Observed 单例状态管理

## 背景

NGA OH 是一个单 entry 模块的鸿蒙原生应用，需要管理论坛 API 请求、本地缓存、用户认证、导航路由等多种状态。初期面临状态管理方案选择。

## 方案对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| `@Observed` 全局单例 ✅ | ArkTS 原生支持、编译无额外依赖、响应式绑定直接 | 大对象变更时可能触发全量通知 |
| 源库状态管理 (Redux 等) | 社区成熟、DevTools 支持 | ArkTS 不支持泛型工具类型、引入外部依赖增加构建复杂度 |
| 组件内全量 @State | 零共享复杂度 | 跨组件数据传递链过长、@Prop 层级太深 |

## 决策

选择 `@Observed` 装饰的 Class 全局单例 + `AppStorage` 的混合方案。

## 理由

1. ArkTS 对泛型和映射类型支持有限，引入 Redux 风格的不可变 reducer 需要大量类型声明，收益不匹配成本。
2. `@Observed` 搭配 `@State` 可以实现粒度的响应式更新，与 ArkUI 渲染引擎的脏检查机制天然契合。
3. 跨页面共享的简单标量（断点、主题名）存入 `AppStorage`，复杂业务状态放在 `@Observed` 类中，分工清晰。

## 代价

- 全局单例模式不利于单元测试隔离
- 大型 `@Observed` 对象的任何字段变更会导致该对象的所有观察者收到通知（需配合不可变替换规避）
